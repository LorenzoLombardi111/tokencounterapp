How to Count Tokens for LLMs — A Practical Guide

Token counting is the difference between a prompt that fits and one that errors out, and between a budget that holds and one that blows up by 10x. This guide explains what tokens are, how every major LLM tokenizes text, and how to count tokens for GPT, Claude, Gemini, Llama, and the rest before you ever hit the API. freetokencounter.app makes the counting itself a single browser tab — but understanding why the count varies across models will save you real money.

What is a token?

A token is the smallest unit of text a large language model processes. It's not a character and it's not always a word — it's a learned subword unit that the model's tokenizer chose during training because that combination of letters appeared often enough to be worth its own slot in the vocabulary. Common short English words like "the" or "and" are one token. Longer or rarer words split into multiple pieces: "tokenization" might be ["token", "ization"], two tokens. Numbers, punctuation, code identifiers, and non-Latin characters all have their own behavior.

The rough rule for English: 1 token ≈ 4 characters ≈ ¾ of a word. So a 1,000-word essay is around 1,330 tokens, and a 500-character tweet is around 125. But the rule is loose — for code, JSON, and non-English text, token counts run 30–100% higher.

Why count tokens?

  1. Cost — APIs bill per million tokens. GPT-4o output costs $10/1M, Claude Opus output is $75/1M. A 5x miscount on a high-volume pipeline is a 5x bill.
  2. Context fit — every model has a hard limit (4K, 128K, 200K, 2M). Exceeding it returns an error or silently truncates.
  3. Cache eligibility — OpenAI's prompt caching activates at 1,024 tokens; Anthropic's at 1,024 too. Below that threshold, no cache discount applies.
  4. Latency — first-token time scales with input size. A 100K-token request can take 30+ seconds before the first response token appears.

How tokenizers actually work — BPE in 60 seconds

Most modern LLMs use byte-pair encoding (BPE) or a variant. The algorithm:

  1. Start with every byte (0–255) as a base "token."
  2. Find the most frequent adjacent pair across the training corpus and merge them into a new token.
  3. Repeat ~100,000 times.
  4. The resulting vocabulary contains common short sequences ("the_", " is", "ing") as single tokens, while rare sequences fall back to byte-level splits.

GPT-4 uses cl100k_base (~100K tokens). GPT-4o uses o200k_base (~200K). Llama 3 uses its own ~128K BPE. Each tokenizer was trained on different data, so the same input produces different counts. That's why the same prompt costs different amounts on different models, and why you need a multi-model counter to compare.

Token counts vary — here's the typical range

The same 1,000-character English paragraph, tokenized by different models:

GPT-4o (o200k_base): ~190 tokens GPT-3.5 (cl100k_base): ~210 tokens Claude 3.5 Sonnet: ~215 tokens (estimate) Gemini 1.5 Pro: ~180 tokens (estimate) Llama 3 / Llama 3.1: ~205 tokens Mistral Large 2: ~210 tokens DeepSeek V3: ~225 tokens (smaller vocab)

The spread on English is modest — about 20% between the most and least efficient. On code, JSON, and non-Latin scripts, the spread can hit 50%+.

Special cases to watch

Code

Code uses 30–50% more tokens than equivalent prose. Identifiers like getUserById may split into 3-4 tokens. Punctuation density (parens, semicolons, dots) adds tokens fast.

JSON

JSON keys repeat constantly, so caching helps a lot — but uncached, expect 1.5x the tokens of the equivalent prose description.

Non-English text

Latin scripts (Spanish, French, German) are fairly efficient. Chinese, Japanese, Korean, Arabic, and other non-Latin scripts often use 2-3 tokens per character on older tokenizers; newer ones (o200k_base, Gemini) are much better.

System prompts and message overhead

The chat protocol adds 3-7 tokens per message for role markers and separators. freetokencounter.app counts the raw text — add a small overhead for full chat requests.

Frequently Asked Questions

What is a token in plain English?

A token is the smallest unit of text an LLM processes. Most are subword pieces — common short words become one token ("the", "is"), while longer words split into 2-4 pieces ("tokenization" → "token" + "ization"). The rule of thumb: 1 token ≈ 4 English characters or ¾ of a word.

Why do tokens matter for pricing and limits?

LLM APIs bill per token (input + output) and enforce context-window limits in tokens. A 1,000-word email costs about 1,330 tokens. Knowing the count before sending lets you budget exactly and avoid context-overflow errors.

Is a token always one word?

No. Common short words are one token. Longer or rarer words split into multiple tokens. "Hello" is one token; "unbelievably" might be three. Numbers, code, and non-English text use more tokens than equivalent English prose.

How do I count tokens without an API key?

Use freetokencounter.app — it counts tokens for 30+ models in your browser, no sign-up or API key required. For OpenAI specifically, the tiktoken Python library and OpenAI's tokenizer playground both work offline.

What is BPE (byte-pair encoding)?

BPE is the algorithm most modern LLMs use to break text into tokens. It starts with single bytes, then iteratively merges the most frequent adjacent pairs into new tokens. After training, common sequences like "the " or "ing" become single tokens. GPT, Claude, Llama, and Mistral all use variants of BPE.

Are token counts the same across models?

No. Different models train tokenizers on different data with different vocab sizes. The same paragraph might be 200 tokens on GPT-4o, 215 on Claude, and 195 on Gemini. freetokencounter.app shows the difference for any prompt across 30+ models.
